# Advanced Viewer вЂ” Zendesk Custom Views App

A full-featured replacement for Zendesk Support's native Views, built as a private nav-bar app using the Zendesk App Framework (ZAF) v2 and React 18.

---

## Features

- **Views sidebar** вЂ” grouped by Brand, Form, Organization, or Shared/Personal
- **Ticket table** вЂ” sortable columns, date formatting, status/priority badges, tag pills
- **Inline filter** вЂ” real-time search across all visible columns (subject, status, assignee, tags, etc.)
- **Ticket preview** вЂ” hover a row for 600 ms to see a slide-in preview panel (toggleable)
- **Play mode** вЂ” cycle through tickets one-by-one; auto-advances when you submit a ticket
- **Bulk actions** вЂ” Change Status, Assign to Group/Agent, Apply Macro, Add/Remove Tags, Merge, Mark as Spam, Delete
- **Merge** вЂ” pick the target ticket from a confirmation modal before merging
- **Clone View** вЂ” name the clone and open its edit page automatically
- **Delete / Edit View** вЂ” in-app delete with confirmation; direct link to Zendesk Admin edit page
- **Create View** вЂ” shortcut link to Zendesk Admin create-view page
- **Export to CSV** вЂ” exports all currently visible (and filtered) tickets
- **Suspended Tickets** вЂ” dedicated section with Recover / Delete bulk actions
- **Deleted Tickets** вЂ” dedicated section with Restore / Permanently Delete bulk actions
- **Manage Views** вЂ” direct link to the Zendesk Admin views management page

## Local development

### 1. Install dependencies

```bash
npm install
```

### 2. Build the JavaScript bundle

```bash
npm run build
```

This runs webpack in development mode and outputs `assets/bundle.js`.

### 3. Log in to Zendesk CLI

```bash
zcli login -i
```

Follow the prompts to authenticate with your Zendesk account.

### 4. Start the local development server

```bash
zcli apps:server
```

ZCLI starts a local server (default port `4567`) that serves the app files directly from disk. No restart is needed when you change icons, CSS, or `manifest.json` вЂ” just hard-refresh the browser.

### 5. Load the app in Zendesk

Open your Zendesk Support instance and append `?zcli_apps=true` to the URL:

```
https://yoursubdomain.zendesk.com/agent?zcli_apps=true
```

The app will appear in the left navigation rail.

### Development workflow

| What changed | What to do |
|---|---|
| `src/app.js` or `src/index.js` | `npm run build`, then hard-refresh (`Cmd+Shift+R`) |
| Icons, CSS, HTML | Hard-refresh only вЂ” no rebuild needed |
| `manifest.json` | Restart `zcli apps:server`, then hard-refresh |

## Testing

There is no automated test suite. All testing is done by running the app locally via `zcli apps:server` and exercising each feature manually.

**Recommended test scenarios:**

1. **Views list** вЂ” verify views load, counts appear, grouping works
2. **Ticket table** вЂ” open a view, sort columns, check date and status formatting
3. **Filter** вЂ” type in the filter box and confirm status, requester, and tags all match
4. **Bulk actions** вЂ” select multiple tickets and test each action type
5. **Play mode** вЂ” click Play, submit a ticket, confirm it auto-advances
6. **Preview** вЂ” hover a row for ~600 ms, confirm the panel appears; toggle it off and verify hover does nothing
7. **Suspended / Deleted** вЂ” verify the sections load and bulk actions work
8. **Clone View** вЂ” clone a view, confirm the name modal appears and edit page opens
9. **Export** вЂ” export a view, open the CSV in Excel/Sheets and verify encoding

## Deploying to Zendesk

### Option A вЂ” Upload via Zendesk Admin (recommended for first install)

1. Run `npm run package` to generate `app.zip`
2. In Zendesk, go to **Admin Center в†’ Apps and Integrations в†’ Zendesk Support Apps**
3. Click **Upload private app**
4. Upload `app.zip` and follow the prompts

### Option B вЂ” Deploy via ZCLI

**First-time install:**
```bash
npm run package
zcli apps:create
```

**Update an existing install:**
```bash
npm run package
zcli apps:update
```

ZCLI will ask for the app ID on the first update. You can find it in **Admin Center в†’ Apps and Integrations в†’ Zendesk Support Apps**.

### Option C вЂ” Sideload for acceptance testing (no install required)

```bash
zcli apps:server
```

Then visit `https://yoursubdomain.zendesk.com/agent?zcli_apps=true`. This is not a permanent install вЂ” the app disappears when the server stops.

## Security notes

- **No credentials are stored in source code.** The Zendesk subdomain is fetched at runtime from `client.get('currentAccount')` and held only in React state.
- **No external services are called.** All API requests go through ZAF's `client.request()`, which proxies through Zendesk's own API on the authenticated user's behalf.
- **The app is marked `"private": true`** in `manifest.json`, meaning it will never appear in the Zendesk Marketplace.
- **`assets/bundle.js` is gitignored.** Only source files are tracked. Everyone who clones the repo must run `npm install && npm run build` before the app will work.

### What to check before making the repository public

- [ ] Replace `"author"` in `manifest.json` with real or team contact info
- [ ] Confirm `assets/bundle.js` is NOT tracked (`git status` should not list it)
- [ ] Run `git log --all --full-history -- "**/*.env"` to confirm no `.env` files were ever committed
- [ ] Review any branch history for accidentally committed secrets

## Architecture notes

### Why `ticket_sidebar.html` exists

The Play mode auto-advance feature requires listening to the `ticket.submit.always` event, which is only available to apps installed in the `ticket_sidebar` location. The nav-bar iframe cannot receive this event directly.

The solution: a minimal invisible iframe (`1px` height) is loaded in every ticket sidebar. When the agent submits a ticket, it writes a timestamp to `localStorage`. The nav-bar app listens for the `storage` event and advances Play mode accordingly.

This means the app appears in **two** Zendesk locations: `nav_bar` (visible) and `ticket_sidebar` (invisible).

### State management

The app uses React's built-in `useState` and `useMemo` вЂ” no external state library. User preferences (`groupBy`, `previewEnabled`) are persisted to `localStorage`.
